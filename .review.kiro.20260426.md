# Review Report: claude-config Repository

**Date:** 2026-04-26
**Reviewer:** Kiro (AI)
**Scope:** Full repo review — all commands, agents, hooks, scripts, references, and git history
**Goal:** Verify the `/impl`, `/vuln`, and `/upgrade` slash commands will work correctly in Claude Code on real codebases

---

## CRITICAL — Will break real workflows

### 1. `/vuln` NVD agent and Detect agent have a circular dependency (vuln.md, step 3)

The NVD agent returns the affected package name. The Detect agent needs that package name to scan the repo. Both are spawned "simultaneously in a single message." The Detect agent cannot know what to search for until the NVD agent returns.

The partial mitigation ("If the NVD agent cannot determine the package name, use the CVE ID to make a reasonable inference") only covers the NVD failure case, not the fundamental ordering problem.

**Fix:** Either (a) run NVD agent first, then spawn Detect + Baseline in parallel with the package name known, or (b) have the Detect agent scan ALL dependency files and return a full inventory (no package name needed), then match against NVD results in the Merge step. Option (b) is better — it keeps parallelism.

### 2. `subagent_type: "Explore"` is not a valid Claude Code Agent tool type (impl.md, vuln.md)

Phase 2A/2B in `impl.md` and the Detect agent in `vuln.md` use `subagent_type: "Explore"`. Claude Code's Agent tool does not have `"Explore"` as a documented subagent_type. The valid type for custom agents is `"general-purpose"`. If Claude Code doesn't recognize `"Explore"`, these steps fail silently or error out, blocking the entire workflow.

The `risk-planner` and `code-review` agents are correctly invoked via `"general-purpose"` with `model: "opus"`, but the Explore agent uses a non-standard type. `/upgrade` avoids this issue entirely by not using an Explore agent.

**Fix:** Change all `subagent_type: "Explore"` to `subagent_type: "general-purpose"` and add tool restrictions (Read, Glob, Grep, LS only — no Bash, no Edit) in the prompt text.

### 3. `preload-context.sh` stdout injection is fragile (hooks/preload-context.sh)

The hook writes to stdout (`echo "=== Auto-injected project context ==="`), assuming Claude Code will prepend this to the conversation. This works for the current Claude Code version, but the hook doesn't validate the JSON input structure — if the payload schema changes, `python3 -c` silently returns empty string and the hook exits 0 (correct, but the regex match on `$prompt` will fail silently).

**Severity downgrade:** This works today. The triple-fallback (`prompt` / `user_prompt` / `message`) is good defensive coding. No immediate fix needed, but worth a comment noting the contract dependency.

## HIGH — Correctness issues that will cause failures in specific scenarios

### 4. `git diff` doesn't include untracked/new files — code-review agent gets empty diff (impl.md, vuln.md, upgrade.md)

All three commands use `git diff` to capture changes for the Opus code review. But `git diff` doesn't show untracked files. If an implementation creates only new files (no modifications to tracked files), the diff is empty. The `code-review.md` agent says "Refuse to review without a diff" — so the review step would refuse, blocking the workflow.

**Fix:** Use `git add -N . && git diff` to include intent-to-add files, or `git diff HEAD` after staging, or `git diff --cached` after `git add`.

### 5. `/impl` Phase 2A/2B: no error handling if Explore subagent fails or returns empty (impl.md)

The command says "Wait for the agent's response before proceeding. Do not begin writing the plan until the file map is returned." But there's no fallback if the Explore agent returns nothing useful (empty repo, no matching files, agent timeout). The command would stall.

**Fix:** Add a fallback: "If the Explore agent returns no relevant files or fails, proceed with the plan using your own file reads to gather context."

### 6. `/impl` `[SUBSTITUTE: ...]` placeholders may confuse Claude (impl.md, Phase 2A)

The Explore subagent prompt uses `[SUBSTITUTE: the full implementation description from Phase 0 or Phase 1]`. Claude sometimes treats bracket-wrapped text as literal output rather than a substitution instruction.

**Fix:** Replace with a clearer instruction like "paste the full implementation description here."

### 7. `/impl` Phase 2A/2B Explore agent may lack file-reading tools (impl.md)

The Explore agent is spawned as `subagent_type: "Explore"` (see #2) but needs to read files, grep patterns, and list directories. Even if the type is changed to `"general-purpose"`, the prompt doesn't specify which tools the agent needs.

**Fix:** Use `general-purpose` with explicit Read/Glob/Grep/LS tools specified in the prompt.

### 8. `test-baseline.md` doesn't handle npm workspaces (agents/test-baseline.md)

The agent detects framework by scanning the working directory for `pom.xml`, `build.gradle`, etc. In an npm workspace monorepo, `npm test` at the root may not run workspace tests.

**Fix:** Add: "For npm workspaces, check for a `workspaces` field in `package.json` and run `npm test --workspaces` if present."

### 9. `/upgrade` Phase 1 step 3 Agent B assumes `gh` CLI is installed (upgrade.md)

Agent B runs `gh api repos/<owner>/<action>/releases/latest` to resolve GitHub Actions versions. If `gh` is not installed or not authenticated, every call fails. No fallback is provided.

**Fix:** Add a guard: "If `gh` CLI is not available, fall back to `curl -s https://api.github.com/repos/<owner>/<repo>/releases/latest | jq -r .tag_name`" or skip with a warning.

### 10. Missing agent files if `install.sh` hasn't been run (vuln.md, upgrade.md)

Both commands reference `~/.claude/agents/test-baseline.md` with a fallback to `~/.claude/claude-config/agents/test-baseline.md`. If the user cloned the repo but didn't run `install.sh`, both paths may be absent. The test-baseline agent will fail silently.

**Fix:** Add a guard: "If neither path exists, warn the user to run `install.sh` and skip the baseline step."

### 11. `code-review.md` has `Bash` in its tools list — reviewer could modify files (agents/code-review.md)

The agent prompt says "NEVER modify files. The reviewer reads; the caller writes." But having `Bash` in the tool list means the agent could run `sed`, `git checkout`, etc. The hard rule is a prompt-level constraint, not an enforcement.

**Fix:** Remove `Bash` from the code-review agent's tools list. It has Read/Glob/Grep/LS which is sufficient. If it needs `git diff`, the caller should pass the diff as input (which the command already does).

### 12. `/upgrade` Phase 1 step 8: Opus planning happens AFTER user confirms plan (upgrade.md)

The user confirms the overall plan in step 7, then step 8 runs Opus planning for SIGNIFICANT/HIGH-RISK components. The Opus plan may reveal new issues (re-classification, blockers) that invalidate the confirmed plan. This creates a confusing UX: the user already said "go" but now gets asked again per-component.

**Fix:** Run Opus planning BEFORE user confirmation (between steps 6 and 7), so the user sees the full picture including Opus-generated plans before confirming.

### 13. `/upgrade` Phase 2 baseline captured AFTER Agent B already modified files (upgrade.md)

Agent B in Phase 1 step 3 "applies the updates in-place" to `.github/workflows/` files. The baseline test run in Phase 2 step 1 runs AFTER these files have been modified. If workflow changes affect the test suite, the baseline is tainted.

**Fix:** Either (a) capture the baseline BEFORE Agent B applies changes, or (b) have Agent B report changes without applying them, then apply in Phase 2.

### 14. Spring Boot `javax` → `jakarta` migration not documented anywhere (references/upgrade/)

The biggest breaking change in Spring Boot 2→3 is the `javax.*` → `jakarta.*` namespace migration. Neither `compatibility.md` nor `ecosystems.md` mentions it. The Agent A research step would likely discover it from release notes, but having it in the reference would prevent missed migrations.

**Fix:** Add a "Known major migrations" section to `compatibility.md` documenting the `javax` → `jakarta` change for Spring Boot 3.x.

## MEDIUM — Will work but suboptimally or with edge-case failures

### 15. `test-notify.sh` passes `$output` as argv — may hit ARG_MAX (hooks/test-notify.sh)

`python3 - "$command" "$output"` passes the entire test output as a command-line argument. If test output exceeds ARG_MAX (typically 128KB–2MB on Linux, 256KB on macOS), the command fails.

**Fix:** Pass the output via stdin or a temp file instead of as a positional argument.

### 16. `/impl` Phase 4 "simultaneously in a single message" doesn't match Claude Code behavior (impl.md)

Claude Code's subagent spawning is sequential within a single turn. The instruction may confuse Claude into thinking it needs special syntax.

**Fix:** Reword to: "Spawn all three agents. They are independent and can run in any order."

### 17. `/upgrade` Agent B applies changes in Phase 1 (planning-only phase) (upgrade.md)

Phase 1 header says "no files changed" but Agent B is told to "apply the updates in-place." If the user cancels at step 7, the GitHub Actions files have already been modified.

**Fix:** Either (a) have Agent B return proposed changes without applying them, and apply in Phase 2, or (b) add a `git stash` / `git checkout` rollback if the user cancels.

### 18. `classification.md` threshold "4 or more non-test files" may over-trigger Opus (references/model-routing/classification.md)

A simple rename refactor across 5 files would trigger SIGNIFICANT classification and invoke Opus planning + review, wasting tokens on trivial work.

**Fix:** Add a qualifier: "4 or more non-test files changed AND the change involves non-trivial logic modifications (not just renames, import updates, or mechanical refactors)."

### 19. `/vuln` Co-authored-by email may be rejected by corporate git hosts (vuln.md)

The commit template includes `Co-authored-by: Claude Code <noreply@anthropic.com>`. Corporate Bitbucket instances may reject commits with unrecognized email addresses.

**Fix:** Make the `Co-authored-by` line conditional on project convention.

### 20. `install.sh` uses `rm -rf` instead of `rm -f` for symlinks (install.sh)

`remove_our_symlink()` uses `rm -rf "$link"`. For a symlink, `rm -f` suffices and is safer. The `uninstall.sh` correctly uses `rm -f`.

**Fix:** Change to `rm -f`.

### 21. Non-atomic `settings.json` writes in install.sh and uninstall.sh

The inline Python opens `settings.json` for reading, then overwrites it in place. If interrupted mid-write, the file could be left truncated/corrupt.

**Fix:** Use a write-to-temp-then-rename pattern.

### 22. `uninstall.sh` doesn't handle missing `python3` (uninstall.sh)

If `python3` is not in PATH, the settings.json stripping step fails with `set -euo pipefail` causing the entire script to exit non-zero.

**Fix:** Add a `command -v python3 &>/dev/null` guard before the Python heredoc, with a warning message if missing.

### 23. `/vuln` OWASP regex `A\d` only matches single digit (vuln.md, step 2)

The filter regex `OWASP \d{4}:A\d` won't match `OWASP 2021:A01` because `A\d` matches only one digit after A, but OWASP identifiers use two digits (A01-A10).

**Fix:** Change to `A\d+` or `A\d{2}`.

### 24. `/upgrade` no cycle detection for circular companion upgrades (upgrade.md, Phase 2 step 5)

If upgrading A requires upgrading B, and upgrading B requires upgrading A at a different version, there's no cycle detection. The command could loop.

**Fix:** Add cycle detection or a max-depth limit for companion upgrade chains.

### 25. `build-systems.md` missing Ruby section despite detection table listing it (references/fix-vuln/build-systems.md)

The detection table lists 8 ecosystems including Ruby, but there's no `## Ruby` section with detect/update/verify commands.

**Fix:** Add a Ruby/Bundler section.

### 26. `nvd-api.md` safe-version derivation is underspecified (references/fix-vuln/nvd-api.md)

The file says "one patch above `versionEndIncluding`" but doesn't explain how to compute "one patch above" for arbitrary version schemes (e.g., `1.2.3` → `1.2.4`? What about `1.2.3.Final`?). Also, the multiple-range logic oversimplifies — it should match the project's current version line, not just take the highest upper bound.

**Fix:** Add examples for non-standard version schemes and clarify range-matching logic.

### 27. `/vuln` Detect agent scan list narrower than `build-systems.md` coverage (vuln.md, step 3)

The command tells the Detect agent to check `pom.xml, build.gradle, build.gradle.kts, package.json, requirements.txt, go.mod, Cargo.toml` — missing `*.csproj`, `Gemfile`, `composer.json` even though `build-systems.md` covers .NET and the detection table includes Ruby/Bundler.

**Fix:** Align the scan list with `build-systems.md` coverage.

### 28. `ecosystems.md` PHP section missing despite detection table (references/upgrade/ecosystems.md)

The detection table lists `composer.json → PHP → Composer` but there's no `## PHP / Composer` section.

**Fix:** Add a PHP/Composer section or remove from the detection table.

### 29. `install.ps1` uses `Remove-Item -Recurse -Force` before creating link (install.ps1)

If `$LinkPath` is somehow a directory with user content, this silently deletes it. The bash version uses `ln -sf` which only replaces files/symlinks.

**Fix:** Check if the target is a symlink/file before removing, or use a less aggressive removal.

### 30. `uninstall.ps1` copy removal is unconditional (uninstall.ps1)

`Remove-IfOurs` removes non-symlink files unconditionally, assuming they were created by `install.ps1`. If a user manually placed a file at `~/.claude/commands/impl.md`, it would be deleted. The bash uninstaller only removes symlinks pointing into `claude-config/`.

**Fix:** Add a content hash check or a marker file to verify the file was installed by `install.ps1`.

### 31. Inconsistent `set -euo pipefail` across hooks

`notify-done.sh` and `test-notify.sh` have `set -euo pipefail` but claim "must never block Claude." `preload-context.sh` correctly omits it. While `|| true` guards mitigate the risk, the inconsistency means `notify-done.sh` missed the hardening pass that the other two hooks received.

**Fix:** Remove `set -euo pipefail` from `notify-done.sh` and `test-notify.sh` to match `preload-context.sh`.

## LOW — Polish, documentation, minor robustness

### 32. `settings-additions.json` hook entries don't specify timeout

Claude Code hooks support a `timeout` field. Without it, a hung hook could block Claude indefinitely.

**Fix:** Add `"timeout": 5000` (5 seconds) to each hook entry.

### 33. `smoke.sh` doesn't test `install.ps1` / `uninstall.ps1`

No Windows CI coverage at all.

**Fix:** Add a note in README that Windows scripts are manually tested, or add a PowerShell smoke test.

### 34. `build-systems.md` uses `powershell` syntax highlighting for grep commands (references/fix-vuln/build-systems.md)

Several grep examples use ` ```powershell` as the language tag when they're standard bash/grep commands.

**Fix:** Change to ` ```bash`.

### 35. `nvd-api.md` example uses PowerShell but commands run on Linux/macOS (references/fix-vuln/nvd-api.md)

The "Minimal request" section shows PowerShell syntax (`Invoke-RestMethod`). The NVD agent runs in a bash environment.

**Fix:** Add a `curl` example alongside or instead of the PowerShell one.

### 36. `/impl` Phase 0 says "note embedded images" but Claude Code can't view them (impl.md)

If the spec file has `![diagram](./arch.png)`, Claude Code can't actually view that image through a file read.

**Fix:** Change to: "If the file embeds image references, note them as context but acknowledge they cannot be viewed. Ask the user to describe the image content if critical."

### 37. `/upgrade` output table abbreviation `PASS W/RECS` is undocumented (upgrade.md)

**Fix:** Pick one abbreviation and document it, or use the full `PASS WITH RECOMMENDATIONS`.

### 38. README claims "new files automatically linked" but `install.sh` has hardcoded file lists (README.md, install.sh)

The README says "New files added to the repo are automatically linked on the next run." But `install.sh` has hardcoded lists of commands, agents, and hooks. Adding a new file requires editing `install.sh`.

**Fix:** Either make `install.sh` dynamically discover files (e.g., `ls commands/*.md`), or correct the README.

### 39. `risk-planner.md` lists WebFetch/WebSearch tools but never references them in prompt (agents/risk-planner.md)

The tools section includes web tools, but the planning discipline section never mentions when or why the planner would fetch external resources.

**Fix:** Add a note: "Use WebFetch/WebSearch to check release notes, changelogs, or migration guides when planning upgrades or vulnerability fixes."

### 40. `test-baseline.md` missing skipped count parsing patterns (agents/test-baseline.md)

The output template includes `**Skipped**: [n]` but the parse table doesn't give patterns for skipped counts. Maven outputs `Skipped: N`, pytest outputs `N skipped`, Jest outputs `N skipped`.

**Fix:** Add skipped-count parsing patterns to the parse table.

### 41. Spring Boot docs URL pattern may be outdated (references/upgrade/compatibility.md)

Uses `https://docs.spring.io/spring-boot/docs/current/reference/html/...` — Spring's website was redesigned. The old URL may redirect or 404.

**Fix:** Update to the current URL pattern or add a note about potential redirects.
