  Can this repo be converted to a Claude Code plugin?

  Yes, and it's a very clean fit. Your claude-config repo already has the exact building blocks that Claude
  Code plugins expect:

  ┌──────────────────────────────────┬─────────────────────────────────────────┐
  │ Your repo                        │ Plugin equivalent                       │
  ├──────────────────────────────────┼─────────────────────────────────────────┤
  │ commands/*.md                    │ skills/ (or commands/ — both work)      │
  ├──────────────────────────────────┼─────────────────────────────────────────┤
  │ agents/*.md                      │ agents/                                 │
  ├──────────────────────────────────┼─────────────────────────────────────────┤
  │ hooks/ + settings-additions.json │ hooks/hooks.json                        │
  ├──────────────────────────────────┼─────────────────────────────────────────┤
  │ references/                      │ Reference files bundled with the plugin │
  └──────────────────────────────────┴─────────────────────────────────────────┘

  The migration would be:

  1. Add .claude-plugin/plugin.json manifest to the repo root
  2. Move or restructure commands/ → skills/ (each command becomes a <name>/SKILL.md folder), or keep
  commands/ as-is since plugins support flat .md command files too
  3. Convert your settings-additions.json hook entries into hooks/hooks.json format (same JSON structure,
  just a different file location)
  4. The install.sh/uninstall.sh symlink dance becomes unnecessary — claude plugin install handles
  everything
  The main behavioral change: skill names get namespaced. Instead of /impl you'd get /claude-config:impl (or
  whatever you name the plugin). Your subagent invocations referencing ~/.claude/agents/*.md would need to
  use ${CLAUDE_PLUGIN_ROOT}/agents/*.md instead.

  Can you make a private marketplace?

  Yes, absolutely. Claude Code has first-class support for private plugin marketplaces. You'd:

  1. Create a marketplace.json in a git repo (can be the same repo or a separate catalog repo):

  {
    "name": "my-team-tools",
    "owner": { "name": "Your Name" },
    "plugins": [
      {
        "name": "claude-config",
        "source": { "source": "github", "repo": "your-org/claude-config" },
        "description": "Implementation, vuln fix, and upgrade workflows"
      }
    ]
  }

  2. Host it on GitHub (public or private), GitLab, or any git server. Private repos work — Claude Code uses
  your existing git credentials for manual installs, and GITHUB_TOKEN/GITLAB_TOKEN for background
  auto-updates.
  3. Users add it with:

  /plugin marketplace add your-org/plugin-catalog
  /plugin install claude-config@my-team-tools

  4. For team-wide rollout, you can auto-register the marketplace via .claude/settings.json in any project
  repo:

  {
    "extraKnownMarketplaces": {
      "my-team-tools": {
        "source": { "source": "github", "repo": "your-org/plugin-catalog" }
      }
    }
  }

  You can also enforce it via managed settings with strictKnownMarketplaces to lock down which marketplaces
  your org allows, and use enabledPlugins to auto-enable specific plugins.

  The marketplace can host multiple plugins — so you could package your claude-config commands as one
  plugin, your dtctl/dtmgd skills as another, the Atlassian CLI skill as a third, etc., all in one private
  catalog.
