1. 🟢 Dual-mode test-baseline — highest-value addition

Your test-baseline.md only does capture. The test-baseliner skill has a verify mode:

 Mode: "verify"
 Input: baseline from a previous capture
 Output: structured regression report — which tests flipped, which are new failures

Right now your commands tell Claude to "compare" two raw test outputs mentally — unreliable at scale. A structured verify mode with explicit diff output (previously-passing tests that now fail) is far more robust. Suggest adding this to test-baseline.md.

2. 🟢 Extract Phase 4 into impl-maintenance sub-agent

The impl: skill has a separate impl-maintenance sub-agent that accepts a compact summary handoff and handles docs + knowledge + instructions. Your Phase 4 bakes all this into the parent command. Extracting it would:

 - Make Phase 4 independently testable
 - Allow reuse across /vuln and /upgrade (which currently have no equivalent maintenance step)

The upgrade: skill spawns one upgrade-planner per component in parallel (all at once). Your /upgrade Phase 1 spawns two agents total (Agent A + Agent B), not one per component. For repos with 5+ components, this matters.

The upgrade-executor skill explicitly calls out: "auto-fix any test code breakage caused by the new version's API changes." Your /upgrade Phase 2 step 9 says "Fix any failures" but doesn't distinguish test-code fixes from production-code fixes, and doesn't say to explain each test change. Worth adding explicit callout.
